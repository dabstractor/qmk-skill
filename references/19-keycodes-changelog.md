# 19 — QMK per-release keycode changes (keycode migration catalog)

> **When to load this (discovery rule).** This is the **exhaustive** record of every keycode change across QMK releases. Load it ONLY for:
> - **version migrations** ("I'm updating from 0.2x to 0.3x — what breaks?"),
> - an **old keymap that won't compile** on current QMK, or
> - a keycode/alias **not found** in the common-traps table (`references/00 §H` / `references/17 §4`).
>
> For everyday keycode *names*, use `references/04` + the common-traps table — do **not** load this file.
>
> **This file is an index.** The per-release changesets live in [`keycodes-changelog/`](keycodes-changelog/) (one file per release). Don't read them all — read only the versions in your migration window, or use the tool at the bottom of the generated block to dump a whole window as one document.

**How to use it — the `(source, target]` filter.** To migrate a keymap from version **S** to version **T**: for every release **r** with `S < r ≤ T`, read its four change kinds:

| Kind | Meaning | Action |
|---|---|---|
| `deprecated OLD→NEW` | OLD still compiles (via `quantum_keycodes_legacy.h`) but is on a removal timer | rename to **NEW** now |
| `retired OLD` | the legacy shim was deleted; OLD no longer compiles | replace with the **NEW** named at the release where it was *deprecated* |
| `removed` | feature/keycode deleted entirely; no replacement | delete or substitute |
| `added` | new options available | informational only — **not** a migration requirement |

The exact edit list = `{OLD names in the user's keymap.c}` ∩ `{OLD names deprecated/retired/removed across (S, T]}`. Crossing a `retired` boundary is what forces a rename that previously "just worked."

> **Layout additions are summarized, not diffed.** The `extras/` (keyboard-layout) layer only ever *adds* per-layout keycodes (e.g. `BE_AMPR`, `FA_1A`) — it never renames or removes core keycodes — so a layout addition can't break an existing keymap and the verbatim hjson is omitted. Only the one-line "added layouts: …" summary is retained.

Unified, change-driven catalog of keycode changes across QMK firmware releases, covering every file in the repository that defines keycodes. Four complementary layers are tracked, each chosen as the cleanest faithful signal available (source where the generated files are noisy, generated where it is the readable interface):

| Layer | File(s) | Role | Provenance |
|-------|---------|------|-----------|
| Core keycodes | `quantum/keycodes.h` | the `QK_*`/`KC_*` numeric enum definitions | **generated** from `data/constants/keycodes/*.hjson` |
| Helper macros | `quantum/quantum_keycodes.h` | modifier/layer/mod-tap/unicode macros (`LCTL`, `MO`, `LT`, `UM`, …) | **hand-written** |
| Deprecations | `quantum/quantum_keycodes_legacy.h` | `#define OLD NEW` compat shims | **hand-written** |
| Layouts | `data/constants/keycodes/extras/*.hjson` | per-layout keycode definitions | **DD source** (generates `quantum/keymap_extras/*.h`) |

A rename can be split across layers and releases: the new macro lands in `quantum_keycodes.h`, the DD enum in `keycodes.h`, and the `OLD→NEW` alias in the legacy file — sometimes together, sometimes staggered (e.g. `X→UM` at `0.22.0`; the `SGUI`/`ALL_T`/`SAGR` family moved from `quantum_keycodes.h` to the legacy file at `0.30.0`).

## Provenance & methodology (for the migration-guide author)

- **Authoritative source of truth:** `data/constants/keycodes/*.hjson` (data-driven). The generated C files (`keycodes.h`, `keycode_string.*`, `keymap_extras/*.h`) are downstream. `keycodes.h` is tracked here as the readable enum interface; it was verified to capture every enum-level change (the only hjson-only changes are display `label` metadata — which does not affect keymap keycodes and is excluded).
- **Layouts** are tracked via the `extras/` hjson source rather than the generated `keymap_extras/*.h` headers, because the headers are regenerated wholesale (and re-bump copyright) at many patch releases, producing noise; the hjson changes only when a layout is actually added or edited.
- **`quantum/keycode.h` / `keycode_config.*`** define no keycodes (runtime config only).
- **Layout of this catalog:** this index file is hand-written; the block between the `BEGIN GENERATED INDEX` and `END GENERATED INDEX` HTML-comment sentinels below is regenerated by `scripts/gen_keycode_catalog.py`, and each release's changeset is a separate file under `references/keycodes-changelog/`. This split lets an agent load only the versions it needs.
- **Regenerating this file:** from the skill root, run `scripts/gen_keycode_catalog.py <path-to-your-qmk_firmware> -o references/19-keycodes-changelog.md`. It updates the generated block below **in place** (the hand-written prose outside the sentinels is preserved) and (re)writes one file per release under `references/keycodes-changelog/`, pruning any stale ones. The script writes the checkout's canonical GitHub identity (derived from its `origin` remote), **never** a local filesystem path.
- **Concatenating a migration window:** rather than opening each release file by hand, run `scripts/keycodes_migration.py --from <from> [--to <to|latest>]` to print every release changeset in the half-open interval `(from, to]` as a single document (cosmetic-only releases are skipped by default). See the usage block at the bottom of the generated section.

### Migration-window tool — `scripts/keycodes_migration.py`

Given `--from` and `--to`, prints every release changeset in the half-open interval `(from, to]` as one document (cosmetic-only releases are skipped by default):

```
# Everything that changed between two versions, as one dump:
scripts/keycodes_migration.py --from 0.22.0 --to 0.30.0
# From a version up to the newest cataloged release (--to defaults to latest):
scripts/keycodes_migration.py --from 0.25.0
# Just list which release files fall in the window:
scripts/keycodes_migration.py --from 0.22.0 --to 0.30.0 --list
# Keep copyright/whitespace-only releases too (skipped by default):
scripts/keycodes_migration.py --from 0.22.0 --include-cosmetic
```
