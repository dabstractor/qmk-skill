# 14 — QMK Configurator, the Compile API, and VIA/Vial

> **Scope.** This reference covers three adjacent QMK-adjacent systems: (1) **QMK Configurator** — the web UI for building keymaps; (2) the **QMK API** (the asynchronous compile API that powers Configurator and any third-party GUI); and (3) **VIA & Vial** — runtime keymap editors that talk to a flashed keyboard over HID.
>
> **Relationships.**
> - **`03-config-and-info-json.md`** — `info.json` (esp. the `layouts.LAYOUT.layout` KLE-derived array and `usb` VID/PID) is what makes a keyboard *appear* in Configurator and is what VIA enumerates against. The two are tightly coupled.
> - **`02-getting-started-build.md`** — `qmk compile`/`qmk flash` accept the *same* `keymap.json` the API accepts; `qmk json2c` and `qmk c2json` convert between `keymap.json` ↔ `keymap.c`. The API uses the same `json2c` machinery server-side.
> - **`10-connectivity.md`** — VIA/Vial ride on the **rawhid** feature (`RAW_ENABLE`, usage page `0xFF60`). The rawhid transport is documented there; this file covers the *VIA/Vial protocol layer on top of it*.
> - **`15-flashing-debugging.md`** — the bootloader question (`.hex` vs `.bin`, halfkay vs caterina vs dfu vs qmk-dfu) determines what file the API/Configurator output you can flash.

## 1. QMK Configurator — the web UI

### Summary
QMK Configurator (<https://config.qmk.fm>) is a single-page application (SPA) that lets a user pick a supported keyboard, lay out a keymap in a visual grid, export the keymap as `keymap.json`, and request a server-side compile that returns a flashable firmware binary (plus matching source, for GPL compliance). It is the no-toolchain path: the user never runs `make` or installs QMK MSYS / the CLI.

### Architecture: frontend → API → compiler
Configurator is a thin client. The real work happens in a small fleet of services:

```
 Browser SPA (qmk_configurator, on GitHub Pages)
        │  (1) pulls keyboard/layout metadata,   (2) POSTs keymap.json
        ▼
 Keyboard Metadata (https://keyboards.qmk.fm, JSON on DigitalOcean Spaces)
   ─ generated by a qmk_firmware CI job (`.github/workflows/api.yml`) every time
     a keyboard changes; Configurator builds its UI from this.
        │
        ▼
 QMK API  (http://api.qmk.fm)  — Flask app
   ─ POST /v1/compile  → enqueue
   ─ GET  /v1/compile/<job_id>  → poll status
   ─ GET  /v1/compile/<job_id>/download  → firmware
   ─ GET  /v1/compile/<job_id>/source    → source zip
        │  (Redis / RQ queue)
        ▼
 QMK Compiler worker (qmk_compiler)
   ─ fresh checkout of requested qmk_firmware branch
   ─ builds keymap.c from keymap.json
   ─ runs `qmk compile keymap.json`
   ─ zips source, uploads firmware + source + metadata to S3-compatible Spaces
```

**Repositories** (each is separate):
| Component | Repo | Role |
|---|---|---|
| Configurator SPA | `qmk/qmk_configurator` | frontend; deployed to GitHub Pages on push to `master` |
| Keyboard metadata generator | `qmk/qmk_firmware` CI | `api.yml` workflow emits JSON metadata to Spaces |
| QMK API service | `qmk/qmk_api` | Flask app, Redis/RQ, S3 |
| QMK Compiler worker | `qmk/qmk_compiler` | the actual `qmk compile` runner |
| Web stack (dev) | `qmk/qmk_web_stack` | docker-compose to self-host API + workers + Redis + S3 |

### How a compile job flows (API status states)
The API is **asynchronous**: you POST a job, then poll. There are **3 non-error Configurator-visible states** ("Waiting for an oven", "Baking", "Finished") mapping to the API's 5 raw statuses:

| API `status` | Meaning |
|---|---|
| `queued` | Job accepted, no worker free yet |
| `running` | Worker has it; `qmk compile` in progress |
| `finished` | Done — inspect `result` for download URLs |
| `failed` | The compile *service* broke (not a user keymap error) |
| `unknown` | Serious internal error; file a `qmk_compiler` bug |

The browser polls roughly every 2 s (or longer). Finished `result` keys: `firmware_binary_url`, `firmware_keymap_url` (the generated `keymap.c`), `firmware_source_url` (full source zip), `output` (stdout/stderr — compile errors show up here).

### Step-by-step (user workflow in the SPA)
1. **Select keyboard** — dropdown filtered by the metadata store. **Select the right revision/subkeyboard.** If the board is not listed, Configurator cannot build for it at all (see §3).
2. **Select layout** — pick the `LAYOUT_*` macro matching your physical build; fall back to `LAYOUT_all` if none matches exactly.
3. **Name the keymap** — must not collide with a name already in the firmware repo, or compile may be refused. Renaming is the first thing to try on a compile failure.
4. **Define the keymap** — drag-drop, click-then-click-keycode, or click-then-press-physical-key. Leave unused positions blank; if a key (e.g. split backspace) appears twice in `LAYOUT_all` but once physically, put the same keycode in both.
5. **Save** — `Download this QMK Keymap JSON File` produces a Configurator `keymap.json`. Reload it later with `Upload a QMK Keymap JSON File`.
6. **Compile** — green **Compile** button, then green **Download Firmware**.

> **JSON format caveat:** the Configurator `keymap.json` is *not* the format used by `kbfirmware.com` or other legacy tools. Mixing them breaks everything. See §2 for the exact schema.

## 2. The `keymap.json` format (Configurator + API native)

This is the single interchange format between the SPA, the API, `qmk compile`/`qmk flash`, and `qmk json2c`/`qmk c2json` (see `02-getting-started-build.md`).

### Required keys (for a Configurator *default* keymap submission)

| Key | Type | Notes |
|---|---|---|
| `keyboard` | string | Full path as used by `make`, e.g. `"hineybush/h87a"`, `"clueboard/66/rev2"` |
| `keymap` | string | Name; should be `"default"` for a repo default keymap |
| `layout` | string | The `LAYOUT_*` macro name, e.g. `"LAYOUT_all"` |
| `layers` | array of array of string | One array per layer; each keycode as a string; length must equal the layout macro's key count |

Optional / bookkeeping:

| Key | Type | Notes |
|---|---|---|
| `commit` | string | **Not consumed by the API.** Used by Configurator maintainers to track which `keymap.c` commit a JSON was generated from. SHA from `git log -1 --pretty=oneline -- keyboards/<kb>/keymaps/default/keymap.c` |

### Example
```json
{
  "keyboard": "hineybush/h87a",
  "keymap": "default",
  "commit": "ef8878fba5d3786e3f9c66436da63a560cd36ac9",
  "layout": "LAYOUT_all",
  "layers": [
    ["KC_ESC","KC_F1","KC_F2","KC_F3","KC_F4","KC_F5","KC_F6","KC_F7","KC_F8","KC_F9","KC_F10","KC_F11","KC_F12","KC_PSCR","KC_SCRL","KC_PAUS"],
    ["KC_TRNS","KC_TRNS","UG_TOGG","QK_BOOT","KC_TRNS","KC_TRNS"]
  ]
}
```
Whitespace inside `layers` arrays is cosmetic only.

### json2c / c2json mapping (cross-ref `02`)
Server-side the API runs `json2c` to turn `keymap.json` into a `keymap.c` that contains exactly one `keymaps[][MATRIX_ROWS][MATRIX_COLS]` array wrapped in the chosen `LAYOUT` macro — nothing else. There is **no place for custom C** (see Gotchas). The reverse, `c2json`, parses a `keymap.c` back into `keymap.json` so long as the C only contains a `LAYOUT(...)` keymap.

### Gotchas — `keymap.json`
- **Layers must be referenced by numeric index, not by `enum`/`#define` name.** `MO(_FN)` is illegal; you must write `MO(2)`. The format carries no C symbols.
- **No custom code of any kind.** Anything requiring a function in `keymap.c` (Tap Dance, Unicode, `process_record_user`, custom `SEND_STRING`) is **impossible** in Configurator. Worse: if `TAP_DANCE_ENABLE = yes` (or similar) is set at the *keyboard* level in the firmware repo, Configurator **cannot compile any firmware at all for that keyboard** until it is removed. This is a limit of both the API and the JSON spec.
- **Custom keycodes are partially possible** — but only if the keycode is declared at the *keyboard* level (`<keyboard>.h` as `enum keyboard_keycodes { X = QK_KB_0, ... }`) with its logic in `process_record_kb()` in `<keyboard>.c` (which must end by calling `process_record_user()`). Keymap-level `SAFE_RANGE` custom keycodes are invisible to Configurator.
- **`.json` is not portable across tools.** Configurator JSON ≠ kbfirmware JSON ≠ KLE raw data. See §3 for KLE → `info.json` conversion (a *different* JSON).

## 3. Supporting a keyboard in Configurator (reference_configurator_support)

### Summary
A keyboard appears in Configurator **only** because of two pieces of metadata that the firmware CI harvests and publishes:
1. The `LAYOUT(...)` macro in `keyboards/<keyboard>/<keyboard>.h` — defines the *electrical* matrix (the `k<row><col>` → matrix-position mapping).
2. The `layouts.LAYOUT.layout` array in `info.json` — the *physical* layout (KLE-derived x/y/w/h per key).

The two are matched **positionally, by sequence**: the Nth macro parameter maps to the Nth object in the `layout` array. Cross-ref `03-config-and-info-json.md` for the full `info.json` schema; this section is the Configurator-specific view.

### The macro ↔ physical map (worked example: 17-key numpad)
`numpad.h`:
```c
#pragma once
#define LAYOUT( \
    k00, k01, k02, k03, \
    k10, k11, k12, k13, \
    k20, k21, k22,      \
    k30, k31, k32, k33, \
    k40,      k42       \
  ) { \
    { k00, k01, k02, k03 }, \
    { k10, k11, k12, k13 }, \
    { k20, k21, k22, KC_NO }, \
    { k30, k31, k32, k33 }, \
    { k40, KC_NO, k42, KC_NO }  \
}
```
Corresponding `info.json` (the `layout` array order matches the macro parameter order):
```json
{
  "keyboard_name": "Numpad",
  "maintainer": "qmk",
  "tags": { "form_factor": "numpad" },
  "layouts": {
    "LAYOUT": {
      "layout": [
        {"label":"Num Lock","x":0,"y":0},
        {"label":"/","x":1,"y":0},
        {"label":"*","x":2,"y":0},
        {"label":"-","x":3,"y":0},
        {"label":"7","x":0,"y":1},
        {"label":"8","x":1,"y":1},
        {"label":"9","x":2,"y":1},
        {"label":"+","x":3,"y":1,"h":2},
        {"label":"4","x":0,"y":2},
        {"label":"5","x":1,"y":2},
        {"label":"6","x":2,"y":2},
        {"label":"1","x":0,"y":3},
        {"label":"2","x":1,"y":3},
        {"label":"3","x":2,"y":3},
        {"label":"Enter","x":3,"y":3,"h":2},
        {"label":"0","x":0,"y":4,"w":2},
        {"label":".","x":2,"y":4}
      ]
    }
  }
}
```
| Per-key object key | Meaning |
|---|---|
| `x`, `y` | position in key units (1u = 19.05 mm) from top-left |
| `w`, `h` | width/height in key units (default 1) |
| `label` | debugging aid only — **not displayed** by Configurator |

### Authoring flow: KLE → `info.json`
1. Build the layout in [Keyboard Layout Editor](https://www.keyboard-layout-editor.com/).
2. Copy the **Raw Data** tab.
3. Paste into the [QMK KLE-JSON converter](https://qmk.fm/converter/) → produces the `layouts.LAYOUT.layout` JSON.
4. Save as `info.json` next to `numpad.h`.

### Gotchas — configurator support
- **`KC_NO` vs `XXX` shorthand.** In the `.h` macro, `KC_NO` marks absent matrix positions. Boards often `#define XXX KC_NO` for readability. **This differs from keymap conventions** which use `XXXXXXX` (7 X's) for `KC_NO` and `_______` (7 underscores) for `KC_TRNS`. Prefer bare `KC_NO` to avoid user confusion.
- **The Configurator does not render key rotation or non-rectangular shapes.** ISO Enter is conventionally drawn as a 1.25u × 2u rectangle right-aligned to the alphanumerics.
- **Vertically-offset keys confuse the KLE converter** (the classic case: arrow keys on 1800/96%-style boards such as TKC1800). Build them un-offset in KLE, then hand-edit the `y` values (e.g. `y:5` → `y:5.25`) in the generated JSON.
- **Macro order must equal `layout` array order.** If you reorder the macro parameters you *must* reorder the JSON identically or every key will be mislabeled.
- **Board must be on `master` of `qmk/qmk_firmware`.** PRs on other branches, or keyboards only in a vendor fork, do **not** show up. See §5 troubleshooting.

## 4. The QMK Compile API (using + self-hosting)

### Summary
The same JSON that Configurator uses is a public REST endpoint anyone can call to compile firmware for any QMK keyboard without a local toolchain. Useful for building your own GUI/CLI, CI, or "compile-on-the-cloud" flows.

### Endpoints

| Method | Path | Purpose |
|---|---|---|
| POST | `/v1/compile` | Submit `keymap.json`; body is the JSON document. Returns `{enqueued, job_id}` |
| GET | `/v1/compile/<job_id>` | Poll status (`queued`/`running`/`finished`/`failed`/`unknown`) and, when finished, `result` |
| GET | `/v1/compile/<job_id>/download` | Download the compiled firmware binary (redirects to an authenticated Spaces URL) |
| GET | `/v1/compile/<job_id>/source` | Download the source zip |
| GET | `https://keyboards.qmk.fm/v1/constants_metadata.json` | "Locked-in" constant versions (keycodes today; more subsystems over time) |
| GET | `https://keyboards.qmk.fm/v1/constants/<subsystem>_<version>.json` | The actual constants, e.g. `constants/keycodes_0.0.1.json` |

> The `master`/`develop` split: `keyboards.qmk.fm/v1/...` is `master` (locked); `keyboards.develop.qmk.fm/v1/...` is `develop` (extra versions there are subject to change).

### Worked example
```bash
# 1. Submit
$ curl -H "Content-Type: application/json" -X POST \
       -d "$(< json_data)" https://api.qmk.fm/v1/compile
{ "enqueued": true, "job_id": "ea1514b3-bdfc-4a7b-9b5c-08752684f7f6" }

# 2. Poll (every ~2 s)
$ curl https://api.qmk.fm/v1/compile/ea1514b3-bdfc-4a7b-9b5c-08752684f7f6
{ "status": "running", "result": null, ... }

# 3. When status == "finished", result contains:
#    firmware_binary_url, firmware_keymap_url, firmware_source_url, output
```
The `output` field carries `qmk compile` stdout/stderr — **compile errors from your keymap show up here**, not as an HTTP failure.

### Self-hosting (development)
- Start with the **`qmk/qmk_web_stack`** repo — a docker-compose bundle that brings up the API service (Flask), Redis + RQ, S3-compatible storage, and the compiler worker(s).
- The **API service** (`qmk/qmk_api`) is a small Flask app. The four meaningful routes are exactly the `/v1/compile*` endpoints above.
- The **Compiler worker** (`qmk/qmk_compiler`) pulls a job from RQ, checks out the requested `qmk_firmware` branch fresh, runs `qmk compile keymap.json`, zips the source, and uploads firmware + source + metadata to S3, then reports status to RQ.
- "API Clients interact exclusively with the API service" — clients never touch Redis or S3 directly; the API proxies the downloads.

### Gotchas — QMK API
- **Asynchronous by design.** No long-lived connection; you must poll. Don't poll faster than ~2 s.
- **GPL side-by-side rule.** The API always offers source *and* binary together; if you build a client, keep that pairing.
- **Validation is "very basic."** The API does minimal JSON validation before enqueueing — bad keymaps fail at the *worker*, surfaced in `output`, not as an enqueue error.
- **Custom-keycode rule from §2 applies server-side too.** Anything needing C is unbuildable here regardless of who calls the API.

## 5. Configurator / API troubleshooting (consolidated)

| Symptom | Likely cause / fix |
|---|---|
| **Keyboard not in the dropdown** | Board is not on `master` of `qmk/qmk_firmware`, or its `info.json` lacks a valid `layouts.LAYOUT.layout` (§3). Track open keyboard PRs; otherwise encourage the vendor to add support. Nothing Configurator can do client-side. |
| **Wrong/extra physical positions** (e.g. three spacebar slots) | Fill all duplicate physical positions with the same keycode (Space, Backspace, Shift). Or pick a tighter `LAYOUT_*` that matches your build. |
| **Layout mismatch (some keys not where expected)** | The `LAYOUT` macro parameter order ≠ the `info.json` `layout` array order (§3). Fix at the keyboard level, not in Configurator. |
| **Compile fails in-browser** | (a) Random stray keycodes on other layers — re-check every layer. (b) Keymap name collides with one already in the repo — **rename the keymap**. (c) Keyboard has `*_ENABLE = yes` at keyboard level for a feature requiring C (e.g. `TAP_DANCE_ENABLE`) — unbuildable in Configurator until removed (§2). |
| **`.json` rejected / behaves oddly** | You used a non-Configurator JSON (kbfirmware, etc.). Re-export from Configurator. |
| **`.hex` vs `.bin` won't flash** | The API/Configurator emits the format the keyboard's bootloader expects; this is a *flashing* problem, not a Configurator one. AVR/halfkay/caterina typically want `.hex`; many ARM/STM32 (DFU, UF2) want `.bin` or `.uf2`. See `15-flashing-debugging.md` for the bootloader → file-type matrix. |
| **Other compile errors** | Inspect the finished job's `result.output` (stdout+stderr). Treat it like local `make` output. |
| **Service-side compile failure (`status: failed`)** | The *service* broke, not your keymap. Retry; file a `qmk_compiler` issue if persistent. |
| **Bugs / feature requests** | File at `qmk/qmk_configurator` (UI) or `qmk/qmk_compiler` (build) — pick the right repo. |

## 6. rawhid — the transport both VIA and Vial sit on

(Cross-ref `10-connectivity.md` for the full rawhid reference; this is the Configurator/VIA-relevant slice.)

| Setting | Default | Notes |
|---|---|---|
| `rules.mk`: `RAW_ENABLE = yes` | off | Adds the raw HID interface and compiles `raw_hid.c` |
| `config.h`: `RAW_USAGE_PAGE` | `0xFF60` | Vendor-defined usage page — **this is the page VIA listens on** |
| `config.h`: `RAW_USAGE_ID` | `0x61` | Usage ID on that page |

C API (implement/call in `keymap.c` or keyboard code):
```c
void raw_hid_receive(uint8_t *data, uint8_t length); // host→keyboard callback; length always RAW_EPSIZE (32)
void raw_hid_send(uint8_t *data, uint8_t length);    // keyboard→host; buffer must be 32 bytes
```
**Hard constraint:** every report in both directions is exactly `RAW_EPSIZE` (**32 bytes**) — HID has no variable-length reports. VIA/Vial build their own multi-report framing on top of this fixed 32-byte pipe. `RAW_ENABLE` is auto-forced on by `VIA_ENABLE` (see §7), so you don't set it yourself when using VIA.

## 7. VIA & Vial (Synthesized — not from QMK docs)

> ⚠️ **Scope of this section.** The QMK docs above cover only QMK Configurator and the QMK Compile API. VIA and Vial are *separate projects* that build on QMK's rawhid feature (§6, `10-connectivity.md`) but have their own repos, apps, and protocols. **Vial's build mechanics, file layout, and force-enabled feature bundle (§7.5) are verified directly from `vial-kb/vial-qmk`** (`builddefs/build_vial.mk`, `quantum/vial.c`, real keymaps). **VIA's bundled protocol code (§7.2, §7.7) is verified from upstream QMK's own tree.** Only the *byte-level protocol framing* and the *VIA app's UI behavior* remain best-effort — confirm those against the upstream VIA (`the-via/app`, `the-via/keyboards`) and Vial (`vial-kb/vial-qmk`, `vial-kb/vial-gui`, `vial-kb/docs`) sources before relying on them.

### 7.1 What VIA is
**VIA** is a web app (and Electron desktop app) that edits a *flashed* keyboard's keymap **at runtime over USB**, without reflashing. It opens the board's rawhid interface (usage page `0xFF60`, usage `0x61` — exactly QMK's `RAW_USAGE_PAGE`/`RAW_USAGE_ID` defaults from §6) and speaks the VIA protocol to read/write the dynamic keymap, lighting, and a few other knobs stored in EEPROM. Cross-ref `10-connectivity.md` for the rawhid transport.

Key idea: with VIA compiled into firmware, **layers and lighting live in EEPROM and can be rewritten by the app**; you no longer need to recompile to change a key.

### 7.2 Enabling VIA in QMK (grounded + best-effort)
**`rules.mk`:**
```make
VIA_ENABLE = yes
```
What QMK's own build system does when you set this (verified in `builddefs/common_features.mk`):
```make
ifeq ($(strip $(VIA_ENABLE)), yes)
    DYNAMIC_KEYMAP_ENABLE := yes   # EEPROM-backed keymap API
    RAW_ENABLE            := yes   # the rawhid transport (§6)
    BOOTMAGIC_ENABLE      := yes   # so the app can jump to bootloader
    TRI_LAYER_ENABLE      := yes
    ifeq ($(strip $(VIA_INSECURE)), yes)
        OPT_DEFS += -DVIA_INSECURE  # disables the security check (see Vial §7.5)
    endif
endif
```
So **`VIA_ENABLE = yes` implicitly turns on rawhid, dynamic keymap, bootmagic, and tri-layer.** You do *not* separately set `RAW_ENABLE`.

**Bundled protocol code** (verified present in QMK tree): `quantum/via.c` and `quantum/via.h` implement the VIA command handler on top of rawhid. Notable constants from `quantum/via.h`:
```c
#define VIA_PROTOCOL_VERSION  0x000D   // = 13; the protocol version compiled into fw
#define VIA_FIRMWARE_VERSION  0x00000000   // unless the keyboard overrides it
```
and a fixed `id_*` command set (`id_get_protocol_version`, `id_dynamic_keymap_get_keycode`/`set_keycode`, `id_dynamic_keymap_get_layer_count`, `id_dynamic_keymap_macro_*`, `id_eeprom_reset`, `id_bootloader_jump`, `id_custom_set_value`/`get_value`/`save`, …) — this is the dynamic-keymap / macro / custom-value API the app drives.

**Layout macro requirement** (best-effort): VIA needs a real `LAYOUT(...)` macro defined for the keyboard (the same macro Configurator uses, §3) because the dynamic keymap is addressed by *(layer, row, column)* against that macro's matrix. A board with no / broken layout macro cannot be VIA-edited.

**Per-keymap files** (best-effort): a VIA-enabled keyboard typically ships a `keymaps/via/keymap.c` (and/or `keymap.json`) so `make <kb>:via` produces a VIA-capable build; the *initial* layers are baked in, then rewritten at runtime into EEPROM.

**Keyboard definition / VIA definitions** (best-effort): the VIA *app* needs to know the keyboard exists — its physical layout, number of layers, encoder/macro counts, etc. This metadata lives in the separate **`the-via/keyboards`** repo ("VIA definitions"). The app ships a snapshot of definitions for known boards; an unknown board must be **side-loaded** via the app's **Design** tab (load a `…via.json` definition file) before it will connect.

### 7.3 The protocol-version gotcha — 🔴 PROMINENT
> **GOTCHA (best-effort, flagged):** The VIA *protocol version* compiled into the firmware (`VIA_PROTOCOL_VERSION`, currently `0x000D` per QMK's `via.h`) must be a version the VIA *app* supports. Mismatches manifest as the app connecting but refusing to edit, showing "unsupported" / "requires a newer VIA," or silently doing nothing. Additionally the keyboard must be **present in the VIA definitions** (or side-loaded via the Design tab) or the app will not recognize it at all even though the firmware is perfectly VIA-capable. Resolution: use a VIA app version new enough for the firmware's protocol version, and make sure the keyboard's VIA definition is loaded.

### 7.4 What's editable at runtime via VIA (best-effort)
- **Layers / keymap** — the whole dynamic keymap (add/remove within the layer count the firmware reports), per-key keycode.
- **Lighting** — for boards with `RGBLIGHT`/`RGB_MATRIX`/`BACKLIGHT`: toggle, hue, saturation, value, effect, speed.
- **Encoders**, **macros** (limited; the `id_dynamic_keymap_macro_*` API), **tap dance / combos** *config* in firmware that exposes them via the custom-value API (varies by board).
- **Bootloader jump, EEPROM reset** — directly from the app.
- **Not** editable: the set of enabled features, keycodes that require C code, anything outside the dynamic-keymap/custom-value surface. (This is exactly the Configurator limit from §2, just applied at runtime.)

### 7.5 What Vial is, and how it differs from VIA

**Vial** is a *separate project* that extends the VIA idea: a cross-platform **`vial-gui`** desktop app (also a web build) editing a flashed keyboard at runtime — but it adds **runtime editing of features VIA cannot touch** and uses a **fork of qmk_firmware** (`vial-kb/vial-qmk`). Vial **requires** `VIA_ENABLE = yes` (it rides the VIA/dynamic-keymap/rawhid transport from §7.2) and layers its own command set on top.

> **Verification note:** the build mechanics, file layout, and force-enabled feature bundle in §7.5.1–7.5.4 are **verified directly from `vial-kb/vial-qmk`** — specifically `builddefs/build_vial.mk`, `quantum/vial.c`, and real `keyboards/*/keymaps/vial/` trees. Only the byte-level protocol framing remains best-effort. (Vial's *prose* docs are NOT in the fork's `docs/` — that folder is just a stale copy of upstream QMK docs; the real docs live in the separate `vial-kb/docs` / get.vial.today.)

#### 7.5.1 Per-keymap files — `keyboards/<kb>/keymaps/vial/`

A Vial keymap needs four files (verified from `keyboards/1upkeyboards/pi60/keymaps/vial/`):

**`rules.mk`**
```make
VIA_ENABLE = yes        # required — Vial rides the VIA transport
VIAL_ENABLE = yes       # Vial-only; valid ONLY in vial-qmk (upstream rejects it, §7.7)
LTO_ENABLE = yes        # typical — Vial adds a lot of code; LTO is usually needed to fit
VIALRGB_ENABLE = yes    # optional — per-key RGB runtime control (§7.5.4)
```

**`config.h`** (the three defines upstream marks `_invalid`, so they must live here and only compile on the fork):
```c
#define VIAL_KEYBOARD_UID {0x09, 0xB9, 0xBD, 0xB8, 0xB8, 0x6A, 0x2A, 0xA7}  // 8-byte, UNIQUE per board
#define VIAL_UNLOCK_COMBO_ROWS { 0, 2 }   // matrix rows of the 2-key unlock combo
#define VIAL_UNLOCK_COMBO_COLS { 0, 13 }  // matrix cols of the 2-key unlock combo
```
Verified these are consumed at runtime: `quantum/vial.c:40-41` read the combo arrays, `quantum/vial.c:95` reads `VIAL_KEYBOARD_UID`. Generate a fresh 8-byte UID per keyboard (`util/vial_generate_uid.py` in the fork, or any random 8 bytes) — **never copy another board's UID**.

**`vial.json`** — the board definition `vial-gui` renders. Consumed **at build time** by `util/vial_generate_definition.py`, which emits `vial_generated_keyboard_definition.h` (a dependency of `quantum/vial.c`, per `build_vial.mk`). Shape:
```json
{
  "lighting": "vialrgb",          // or "none", "raw_rgb", etc. — what lighting vial-gui exposes
  "matrix": { "rows": 5, "cols": 16 },
  "layouts": {
    "labels": [ ["Backspace","2U",...], "Split Left Shift", ... ],  // per-layout-option names
    "keymap": [ /* KLE-style array; each key "r,c\n\n\n<layoutVariantIndex>" maps a
                   physical key to a matrix position AND a layout-option choice */ ]
  }
}
```

**`keymap.c`** — a standard QMK keymap. Vial conventionally uses the **`LAYOUT_all`** macro so every switch position is addressable at runtime; layers are plain QMK layers (Vial reads/writes them via the dynamic-keymap API).

#### 7.5.2 What `VIAL_ENABLE = yes` actually does (verified from `builddefs/build_vial.mk`)

- Compiles `SRC += quantum/vial.c` and defines `-DVIAL_ENABLE -DNO_DEBUG -DSERIAL_NUMBER="vial:f64c2b3c" -DCAPS_WORD_INVERT_ON_SHIFT`.
- **Force-enables a big feature bundle** (so their state is runtime-editable): `TAP_DANCE_ENABLE`, `CAPS_WORD_ENABLE`, `COMBO_ENABLE`, `KEY_OVERRIDE_ENABLE`, `LAYER_LOCK_ENABLE`, `REPEAT_KEY_ENABLE` — and, because `QMK_SETTINGS` defaults to `yes`, also `AUTO_SHIFT_ENABLE`. This is *why* a Vial build is large and *why* you cannot meaningfully disable these per-keymap.
- **`QMK_SETTINGS` (default yes)** is the "QMK Settings" tab in `vial-gui` — runtime-tunable per-key tap-hold behavior. It defines the per-key variants `TAPPING_TERM_PER_KEY`, `PERMISSIVE_HOLD_PER_KEY`, `HOLD_ON_OTHER_KEY_PRESS_PER_KEY`, `QUICK_TAP_TERM_PER_KEY`, `RETRO_TAPPING_PER_KEY`, plus `COMBO_TERM_PER_COMBO`, `CHORDAL_HOLD`, and `FLOW_TAP_TERM=321`. (i.e. Vial flips on most of `04`/`05`'s advanced tap-hold machinery and exposes it live.)

#### 7.5.3 Security model (verified)

Vial gates **secure** operations (rewriting keymap, tap-dance, combos, macros, etc.) behind the per-board **`VIAL_KEYBOARD_UID`** and an **unlock combo** (`VIAL_UNLOCK_COMBO_ROWS`/`_COLS`): the user must hold that 2-key combo to enter an unlocked editing session. To build **without** the secure check (dev boards, unsigned), set:
```make
VIAL_INSECURE = yes     # -> -DVIAL_INSECURE  (note: VIAL_INSECURE, distinct from upstream's VIA_INSECURE)
```

#### 7.5.4 Per-key RGB (verified)

`VIALRGB_ENABLE = yes` compiles `quantum/vialrgb.c` and is paired with `"lighting": "vialrgb"` in `vial.json`. This is Vial's own per-key RGB runtime system (separate from plain `RGB_MATRIX` keycodes) and is what powers the RGB tab in `vial-gui` on supported boards.

#### 7.5.5 What Vial adds over VIA (summary)

- **Runtime editing of:** Tap Dance, Combos, Key Overrides, Macros, Layer Lock, plus a live "QMK Settings" tab (per-key tap-hold tuning) and (with `VIALRGB_ENABLE`) per-key RGB — all absent from stock VIA.
- A native desktop app (`vial-gui`) rather than (only) a web app.
- A security model (per-board UID + unlock combo) by default.
- Cost: a **much larger** firmware (the force-enabled bundle + `vial.c`/`vialrgb.c`/`qmk_settings.c`) — `LTO_ENABLE = yes` is near-mandatory, and flash-tight AVR (atmega32u4) boards may not fit.

#### 7.5.6 Vial rawhid framing (best-effort)

Vial reuses the **same** rawhid endpoint inherited from `VIA_ENABLE` (usage page `0xFF60` / usage `0x61`, 32-byte reports — §6/§7.2) and multiplexes VIA commands and Vial-specific commands (tap-dance/combo/key-override/macro/rgb) over that pipe via a command byte. The exact byte-level framing is best-effort here — confirm against `vial-kb/vial-qmk` `quantum/vial.c` if you're writing a host tool.

### 7.6 The fork-divergence gotcha — 🔴 PROMINENT
> **GOTCHA (best-effort, flagged):** Vial is built on a **fork** of qmk_firmware called **`vial-kb/vial-qmk`**, **not** upstream `qmk/qmk_firmware`. The fork **lags upstream** — QMK features/fixes land in upstream first and are ported into vial-qmk on a delay. Consequences:
> - A keyboard or feature present in upstream QMK may **not yet exist** in vial-qmk.
> - Building a Vial keyboard means checking out **vial-qmk**, not upstream QMK — `qmk setup` against the wrong tree will fail or mis-build.
> - Bug reports / PRs for Vial behavior go to `vial-kb/*`, **not** `qmk/qmk_firmware`.
> - When upstream makes a breaking change, Vial users wait for the fork to catch up.

### 7.7 How QMK's own tree treats VIA vs Vial (grounded)
This is directly observable in `builddefs/common_features.mk` and the data-driven mappings, and is worth knowing precisely:
- **`VIA_ENABLE` IS a valid upstream QMK feature** (`builddefs/generic_features.mk` lists `VIA`), and `builddefs/common_features.mk` wires its side-effects (§7.2). The bundled `quantum/via.{c,h}` are upstream.
- **`VIAL_ENABLE` and the `VIAL_KEYBOARD_UID` / `VIAL_UNLOCK_COMBO_*` defines are NOT.** QMK's `data/mappings/info_rules.hjson` and `info_config.hjson` explicitly mark them as `_invalid`:
  ```
  "VIAL_ENABLE":            {"info_key": "_invalid.vial",        "invalid": true}
  "VIAL_KEYBOARD_UID":      {"info_key": "_invalid.vial_uid",    "invalid": true}
  "VIAL_UNLOCK_COMBO_COLS": {"info_key": "_invalid.vial_unlock_cols", "invalid": true}
  "VIAL_UNLOCK_COMBO_ROWS": {"info_key": "_invalid.vial_unlock_rows", "invalid": true}
  ```
  i.e. upstream QMK refuses to accept these as data-driven config — which is *exactly* why Vial requires its fork (§7.6). If you see a build error complaining about an `_invalid` Vial key while using upstream QMK, that is the fork-divergence gotcha biting: switch to `vial-qmk`.

### 7.8 Firmware-size cost
VIA/Vial are **not free** in flash/RAM:
- `VIA_ENABLE` pulls in rawhid + dynamic keymap + the VIA command handler and an EEPROM-backed layer table; on flash-tight AVR boards (e.g. atmega32u4 28 KB) this can crowd out other features or push past the limit.
- Vial adds *much more* — the force-enabled bundle from §7.5.2 (tap-dance, combo, key-override, macro, layer-lock, repeat-key, auto-shift) **plus** `vial.c`/`vialrgb.c`/`qmk_settings.c`. `LTO_ENABLE = yes` is near-mandatory and many Vial builds target ARM/STM32 precisely for flash headroom.
Exact byte costs are board- and config-dependent; measure with `qmk` size reports (see `15-flashing-debugging.md`).

### 7.9 VIA/Vial quick comparison (best-effort)
| | QMK Configurator / API | VIA | Vial |
|---|---|---|---|
| Edits when? | Compile-time (reflash required) | Runtime, no reflash | Runtime, no reflash |
| Needs recompile to change a key? | **Yes** | No | No |
| Transport | — (server compile) | rawhid `0xFF60`/`0x61` | rawhid `0xFF60`/`0x61` (extra cmds) |
| Custom C in keymap? | **No** | No (runtime only) | No (runtime only) |
| Tap Dance / Combos / Key Overrides editable? | No | Limited / varies | **Yes** |
| Macros editable at runtime? | No | Limited (macro API) | **Yes** |
| Firmware base | upstream `qmk/qmk_firmware` | upstream `qmk/qmk_firmware` | **fork `vial-kb/vial-qmk`** |

| App | web (config.qmk.fm) | web/desktop (the-via/app) | `vial-gui` desktop/web |

## 8. Cross-cutting cheat sheet

- **"Where does my board show up?"** — Configurator and VIA both key off `info.json` metadata: Configurator uses `layouts.LAYOUT.layout` for rendering and `usb` for nothing display-side; VIA enumerates by `usb` VID/PID + the `0xFF60`/`0x61` rawhid interface and needs a layout macro. (`03-config-and-info-json.md`.)
- **"Why won't it compile in Configurator?"** — almost always: custom-C feature enabled at keyboard level, stray keycodes, or a keymap-name collision (§2, §5).
- **"Configurator .json vs my local `qmk compile`"** — identical format; you can `qmk compile keymap.json` locally for the same result without the cloud round-trip (`02-getting-started-build.md`).
- **"VIA connected but won't edit"** — protocol-version mismatch or missing keyboard definition (§7.3).
- **"Vial build fails in upstream QMK with an `_invalid` Vial key"** — you are on the wrong tree; use `vial-qmk` (§7.6, §7.7).
- **"`.hex` vs `.bin`"** — a flashing concern, not Configurator's; the API emits what the bootloader wants (`15-flashing-debugging.md`).
